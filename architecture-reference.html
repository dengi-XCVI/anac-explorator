Architecture reference

ANAC Explorator

A CLI-first, artifact-driven local warehouse for ANAC open procurement data. The system resolves remote CKAN datasets, downloads and caches raw resources, builds schema and semantic artifacts, materializes analytical data into Parquet plus DuckDB metadata, and exposes the result through a stable anacx command surface.

Executable: anacx Primary storage: Parquet + DuckDB Shape: CLI / automation first Maturest family: monthly CIG

1. System overview

The repository is best understood as a local data-product pipeline wrapped in a stable CLI. It is not a web application and it is not just a loose set of scripts. The core idea is to turn ANAC open-data resources into a repeatable, queryable local warehouse while preserving enough artifact lineage that both people and automation can inspect what happened.

Source side

Remote data starts as CKAN package metadata plus downloadable CSV or JSON resources exposed by the ANAC portal.

Artifact side

Raw downloads, schema mappings, vocabulary crosswalks, dictionaries, and comparisons are stored as explicit files on disk.

Warehouse side

Analytical payload lands in Parquet while DuckDB stores control-plane metadata and registered views.

CLI side

Users interact through a stable command set: datasets, download, schema, query, stats, update, config, and drop.

Short architectural statement

ANAC Explorator is a CLI-first local warehouse interface that resolves ANAC datasets into logical families, downloads them into a manifest-backed raw cache, enriches them with schema and semantic artifacts, materializes analytical data into Parquet, and rebuilds a DuckDB metadata layer that the CLI can query safely and consistently.

2. Mental model

Control plane

The parts that decide what should happen.

  • CLI parsing and handler dispatch in cli.py
  • Config merging and path resolution in config.py and paths.py
  • Stable error and output contract in errors.py, models.py, and output.py
  • Dataset-family registry and selection logic in catalog.py and selection.py
  • Metadata view rebuilding in metadata_views.py

Data plane

The parts that move or persist actual dataset content.

  • Remote CKAN fetches via ckan.py and browser.py
  • Manifest-backed downloads via sample.py
  • Schema mapping and comparisons via schema.py and comparison.py
  • Vocabulary and dictionary artifacts via vocabulary.py and dictionary.py
  • Parquet loading and local queries via loader.py

Why this split matters

The repository deliberately separates describing local state from storing analytical payload. DuckDB is used both as a query engine and as the metadata control plane, while Parquet remains the durable data payload layer.

3. End-to-end flow

The main runtime path is consistent across the stable commands, even though each command enters at a different point.

ANAC CKAN + resource endpoints Remote dataset metadata and downloadable CSV/JSON files.
→
Access layer ckan.py resolves package metadata. browser.py provides Playwright transport when the WAF blocks direct HTTP.
→
Registry + selection catalog.py maps logical families to real CKAN datasets. selection.py normalizes year, month, slice, and latest selectors.
Download layer sample.py selects resources, downloads them, extracts archives when needed, and writes a manifest.
→
Raw cache Files land under data/raw/... with manifest.json lineage.
→
Artifact builders Schema, comparison, vocabulary, and dictionary modules generate reusable interpretation artifacts.
Warehouse loader loader.py pushes CSV-heavy work into DuckDB, emits partitioned Parquet, and updates catalog tables.
→
Warehouse state Durable Parquet under data/warehouse/parquet/... plus DuckDB metadata in data/warehouse/anac.duckdb.
→
Metadata discoverability metadata_views.py rebuilds anac_* views from manifests, dictionaries, schemas, loaded resources, and registered views.
Stable CLI surface cli.py exposes datasets, download, schema, query, stats, update, config, and drop.
→
Output contract models.py, output.py, and errors.py normalize JSON/table/YAML output and stable error codes.

Important runtime fact

In this environment, direct HTTP access to ANAC can be blocked by the WAF. The architecture therefore keeps Playwright as a first-class transport path, not as an optional extra.

4. Persistent state surfaces

The local system state is intentionally explicit. Instead of hiding everything inside a single database, the project keeps several durable surfaces with different responsibilities.

Surface Location What lives there Why it exists
Raw resource cache data/raw/... Downloaded archives, extracted materialized files, manifest metadata Preserves source lineage and enables cache reuse or restart-safe downloads
Schema artifacts schemas/*.json Schema mappings and cross-target comparisons Makes schema inspection independent from live re-parsing
Vocabulary artifacts vocabularies/*.json, vocabularies/index.json Crosswalk tables and code-to-label link metadata Supports semantic enrichment and documented joins
Dictionary artifacts dictionaries/*.json, dictionaries/*.md Human-readable field descriptions and semantic references Bridges raw schemas to meaning-oriented documentation
Warehouse payload data/warehouse/parquet/... Partitioned analytical data Keeps large data in a durable, query-friendly format
Warehouse catalog data/warehouse/anac.duckdb Loaded resource records, registered views, period manifest, discoverability inputs Acts as the warehouse control plane and query engine

  • registered_views
  • dataset_period_manifest

Metadata discoverability views

  • anac_dataset_resources
  • anac_partitions
  • anac_registered_views

Config surface

Persisted config lives at ~/.config/anacx/config.json, with compatibility support for the older anac-explorator location.

5. Stable command surface

The canonical interface is the anacx executable. The stable Phase 3 commands are thin entry points over the architectural layers described above.

Command Primary role Main implementation path Main artifacts or views touched
datasets List or inspect logical dataset families and local / remote coverage cli.py → catalog.py anac_datasets, local manifests, registry metadata
download Plan or execute downloads and optional Parquet loads cli.py → selection.py → catalog.py → sample.py / loader.py Raw cache, schemas, Parquet, DuckDB catalog, optional validation
schema Inspect schema artifacts, diffs, or warehouse DDL cli.py → schema_service.py Schema JSON, dictionary JSON, anac_registered_views
query Run safe SQL against local relations and metadata views cli.py → errors.py → loader.py DuckDB relations, anac_* views, read-only policy
stats Summarize local state, partitions, and optional live profiling cli.py → stats.py anac_* views and optionally live target relations
update Plan or apply incremental refreshes cli.py → catalog.py Raw cache, Parquet, DuckDB catalog, vocabulary refresh, dictionary regeneration
config Show, set, unset, reset, and validate effective config cli.py → config.py Config file, env overrides, effective runtime settings
drop Plan or apply safe local pruning cli.py → catalog.py → drop.py Selected raw files, selected Parquet files, reconciled metadata

Compatibility note

The repository still exposes the legacy executable alias anac-explorator and earlier low-level commands, but the stable architecture is organized around anacx and the eight commands above.

6. Component map

The sections below move from the outside of the system inward. Each block explains what the component owns, what it depends on, and what artifacts or interfaces it produces.

CLI entry point and shared command contract

Key files

src/anac_explorator/cli.py, src/anac_explorator/__main__.py, src/anac_explorator/models.py, src/anac_explorator/output.py, src/anac_explorator/errors.py

Responsibility

Parse user intent, bind effective paths and configuration, dispatch to a handler, and return a stable success or error envelope.

Why it matters architecturally

  • Keeps the top-level CLI consistent across all commands.
  • Encodes confirmation rules for destructive or mutating actions.
  • Normalizes JSON, table, and YAML rendering behavior.
  • Turns internal exceptions into stable error codes and exit codes.
Remote access and transport layer

Key files

src/anac_explorator/ckan.py, src/anac_explorator/browser.py

Responsibility

Resolve CKAN package metadata and fetch raw content through either standard HTTP or Playwright-backed browser transport.

Important behavior

  • ckan.py owns package_show resolution and request headers.
  • browser.py is a first-class solution for WAF-protected access.
  • The transport choice is configurable and flows through the CLI.
Config and path resolution

Key files

src/anac_explorator/config.py, src/anac_explorator/paths.py

Responsibility

Merge defaults, persisted config, environment variables, and CLI flags into one effective runtime config and one consistent set of paths.

Architectural value

  • Avoids path and transport logic being reimplemented by each command.
  • Lets commands publish their active filesystem paths in metadata output.
  • Supports both canonical ANAC_* env vars and compatibility names.
Dataset-family registry and temporal selection

Key files

src/anac_explorator/catalog.py, src/anac_explorator/selection.py

Responsibility

Translate user-facing logical families such as cig into concrete CKAN package IDs, resources, capabilities, and update / drop behavior.

Architectural value

  • Separates the public CLI vocabulary from raw CKAN dataset naming.
  • Centralizes capability differences between temporal and snapshot families.
  • Provides the reusable planning contract for download, update, and drop.
Download and raw-cache layer

Key files

src/anac_explorator/sample.py

Responsibility

Select the right resource, download it, materialize extracted content when needed, and write manifest.json so later stages can reason about the resource without guessing.

Architectural value

  • Provides cache reuse instead of blind re-downloads.
  • Supports restart-safe behavior and resumable HTTP where possible.
  • Creates a durable bridge between remote resources and local warehouse state.
Schema, comparison, vocabulary, and dictionary services

Key files

src/anac_explorator/schema.py, src/anac_explorator/comparison.py, src/anac_explorator/schema_service.py, src/anac_explorator/vocabulary.py, src/anac_explorator/dictionary.py

Responsibility

  • Capture raw column structure and inferred types.
  • Compare schema targets over time.
  • Build code-to-label crosswalks from vocabulary datasets.
  • Generate field dictionaries that combine schema and semantic knowledge.

Conceptually, this is the interpretation layer

The raw cache tells you what was downloaded. The interpretation layer tells you what the columns and codes mean. This is why the schema and dictionary artifacts sit beside the warehouse instead of being hidden inside it.

Warehouse loading and local query execution

Key files

src/anac_explorator/loader.py

Responsibility

Keep heavy CSV scanning inside DuckDB, generate Parquet output, register views, and expose local query execution with read-only policy by default.

Architectural value

  • Scales better than Python row-by-row loading for large resources.
  • Maintains a clear distinction between Parquet payload and DuckDB control metadata.
  • Provides a single backend for both loading and querying.
Metadata discoverability layer

Key file

src/anac_explorator/metadata_views.py

Responsibility

Rebuild the full anac_* view layer from local artifacts, warehouse catalog rows, manifests, and dictionaries each time it is needed.

Architectural value

  • Makes the warehouse self-describing.
  • Lets datasets, query, and stats work from a common metadata model.
  • Avoids maintaining a separate permanent metadata warehouse.
Stats, update, integrity, and drop orchestration

Key files

src/anac_explorator/stats.py, src/anac_explorator/catalog.py, src/anac_explorator/integrity.py, src/anac_explorator/drop.py

Responsibility

  • Summarize local state from metadata views.
  • Plan and apply incremental refreshes.
  • Validate local coherence for supported families.
  • Delete selected local artifacts and reconcile catalog state.

7. Detailed data flows

Flow A: dataset discovery
  1. anacx datasets enters through cli.py.
  2. The handler binds effective paths and optional network transport settings.
  3. catalog.py enumerates logical families from the dataset-family registry.
  4. For detailed inspection, the CLI may instantiate ckan.py to enrich remote metadata.
  5. Local coverage is merged from manifests, dictionaries, and warehouse metadata.
  6. The result is returned in the stable command envelope.
Flow B: download to local warehouse
  1. anacx download parses temporal flags into a normalized TemporalSelection.
  2. The dataset-family registry resolves the logical family into a reusable download plan.
  3. sample.py downloads or reuses the raw resource and writes a manifest.
  4. loader.py maps the source into projected DuckDB types and writes Parquet output.
  5. DuckDB catalog rows are updated for loaded resources and registered views.
  6. Optional integrity validation can run after the load for supported families.
Flow C: schema and semantic interpretation
  1. Schema artifacts describe the raw field structure and inferred types.
  2. Comparison artifacts explain cross-target drift over time.
  3. Vocabulary artifacts encode code-to-label mappings and join contracts.
  4. Dictionary artifacts combine schema, comparison, and vocabulary knowledge into field-level documentation.
  5. anacx schema reads these artifacts directly through schema_service.py.
Flow D: querying and stats
  1. anacx query or anacx stats opens the local DuckDB database.
  2. metadata_views.py rebuilds the anac_* views in that session.
  3. query enforces read-only behavior unless the CLI explicitly opts into mutation.
  4. stats either reads metadata summaries or profiles live dataset relations through DuckDB aggregates.
  5. The response is formatted through the shared output contract.
Flow E: update and drop lifecycle
  1. anacx update uses the registry to decide whether a family supports incremental refresh.
  2. For supported families, update planning compares requested scope with the latest local state.
  3. Execution reuses the existing download and load building blocks rather than inventing a separate pipeline.
  4. Vocabulary-aware updates can regenerate the current CIG dictionary when semantic inputs changed.
  5. anacx drop creates a precise deletion plan and removes only the selected raw and/or Parquet files.
  6. After deletion, metadata is reconciled so the local warehouse does not lie about what still exists.

9. Strengths and boundaries

Current strengths

  • The stable top-level CLI contract is fully defined.
  • The local metadata layer is queryable and rebuilt on demand.
  • The storage model is efficient: Parquet payload, DuckDB control plane.
  • Mutation-oriented commands require explicit intent and confirmation.
  • The monthly CIG family is the most complete end-to-end workflow.
  • Vocabulary refresh is wired into the stable update surface.

Known boundaries

  • The project remains CLI-first; there is no web UI.
  • Not every dataset family exposes the same lifecycle depth.
  • Drop support is intentionally strongest for CIG.
  • Update support is broader than before, but still not universal.
  • The architecture is strongest around artifact-driven local analysis rather than a polished Python library API.

This reference reflects the current repository state: a coherent local warehouse interface with explicit artifacts, a stable CLI envelope, and a stronger production-like path for CIG than for every other family.
